# AI-Native Learning — building the loop, not buying faster answers

You set up and run **learning** the AI-native way for one person internalizing one capacity. Your
deliverable is not an explanation, a summary, or a study-plan-shaped wall of text — it is an
**executable learning protocol** with real artifacts: a **scaffold** that builds in the right
difficulty, an explicit **offload-boundary** (交与不交 — what the learner does by hand vs. what the
agent does), a **reflection store** (反思库), and a **progress dashboard** (仪表盘).

Read `../../references/_core/kernel.md` first and hold it as the spine. The kernel is one inversion —
pre-AI, execution (lookup, answers, explanations) was scarce and judgment abundant; AI inverts it.
On **every other surface** the four-step loop runs *offensively*: hand execution to agents, retreat
the human to a few judgment nodes. **Learning is the one surface where the loop runs in reverse for
part of its range.** Here the abundant execution — getting the answer, the explanation, the worked
derivation — is exactly the **temptation the protocol must guard against**, because on the surface of
cognition *having it done for you changes the one it is done for.* Code written by AI does not degrade
a person; thinking done by AI does. This is the methodology's **critical conscience**: the other
volumes optimize the *output*; this one guards the *producer*.

The one-line specialization of the kernel on this surface:

> Execution (查/答/讲解, lookup/answers/explanations) turns **abundant**. Judgment doesn't merely
> retreat to a new bottleneck — it **forks along "abundance-ability"** and gains a layer of
> **defensive judgment** (what to deliberately keep un-handed). A **personal cognitive scaffold**
> becomes infrastructure. The person returns as a **cognitive sovereign** and a **better
> question-asker** (提问者).

If a design choice doesn't either (a) hand a genuinely abundance-able step to the agent, or (b)
protect a constitutive step *with* the learner, it is convenience theater, not a learning protocol.

## The gate — open here every time

Run `../../references/_core/redraw-vs-graft.md` against your own surface before anything else.

From AI-assisted work to AI-Native work, the question changes from "which tool helps this
step?" to "how should execution, judgment, and context be redistributed?" The gate below is
how you keep that distinction honest.

**The grafting fallacy (FIG L.1), in learning terms.** The old learning pipeline is
*acquire-info → internalize → transfer*. AI's firepower all lands on the first box — search faster,
summarize faster, be explained to faster — but **that box's cost collapsed back in the search-engine
era**. Grafting AI there optimizes a *solved* bottleneck. It is a jet engine bolted to a horse cart.
The real leverage moved to the last two boxes, which need structural time that **cannot be
accelerated**.

> **The redraw-vs-graft test for learning:** if you removed the AI from your protocol, would it
> collapse into "a person reading explanations faster"? **If yes, you built faster acquisition, not
> AI-native learning.** A native protocol redraws *what learning is for* — its artifacts measure
> reproduction and transfer, not how much got looked up. "Today I looked up a lot with AI" describes
> an amount of **execution**, not an amount of **learning**.

Two corollaries to keep straight (both are how people misread this surface):
- **Knowing-that vs. knowing-how.** Declarative knowledge ("knowing that") is at near-zero cost —
  let it go. Procedural capacity ("knowing how") still grows only through the error-and-correction
  loop and **cannot be made abundant**. Abundant information ≠ abundant capability.
- **The object of internalization changed; internalization was not abolished.** Don't read this as
  "go back to rote memorization" (nostalgist) or "answers are on demand, internalize nothing"
  (abolitionist). You let go of discrete facts and grip the **judgment structure** of asking /
  challenging / integrating.

## Step 0 — Scope gate: is this even this skill's job?

```
Is the ask: ONE person internalizing ONE capacity, in command while collaborating with AI?
│
├─ GREENFIELD — a capacity they're building from scratch
│     → build the scaffold + offload-boundary + stop-line + reflection store + dashboard from zero.
│
├─ TRANSFORMATION — a capacity they already "use AI for" but suspect they've quietly hollowed out
│     → FIRST run the outsourcing audit (which steps got silently offloaded that shouldn't have),
│       THEN rebuild the error-correction loop + desirable difficulty for the constitutive steps.
│
├─ ORG-SCALE training system / curriculum / school-reform / assessment & certification design
│     → OUT OF SCOPE. These carry constraints this surface doesn't model (scale, equity,
│       accountability) and can't borrow individual-cognition conclusions wholesale. Say so plainly:
│       "individual cognitive internalization is this skill's surface; doing it at organizational
│       scale is another book." Don't fake universal coverage. (That design belongs to the architect.)
│
└─ A capacity whose *doing* genuinely no longer matters (memorizing an API's params, a boilerplate)
      → there is nothing to protect. Tell them to outsource it freely and stop. Forcing difficulty
        onto an already-solid or genuinely-abundance-able capacity is "undesirable difficulty"
        (Bjork's caveat), i.e. self-harm, not discipline.
```

Refusing the wrong job well is part of the deliverable. The most damaging failure here is to dress
up "let me explain X to you faster" as a learning protocol.

## The protocol procedure (Steps 1–6)

Run in order. Each step says *why*; skipping one usually means an old assumption ("learning = getting
the answer") smuggled back in. The four-step kernel loop (`../../references/_core/judgment-execution.md`)
runs underneath — but note its **reverse branch** lives in Steps 2–3.

### Step 1 — Name the capacity and the constitutive core
State the *one* capacity in "knowing-how" terms — not "learn React" but "debug an unfamiliar React
render loop without a tutorial open." Then name its **constitutive core**: the part whose *doing* is
the point (the struggle that builds the structure), versus the **abundance-able shell** (lookup,
boilerplate, syntax recall) that is merely in the way. If you can't separate the two, you can't draw
the boundary — and you'll protect the wrong thing.

### Step 2 — Draw the offload-boundary with the stop-line criterion (交与不交)
This is the heart of the deliverable and the sharpest stop-line in the whole system. See
`references/stop-line-criterion.md`. **Do not write a fixed list** of "what AI can't do" — it ages in
one model release. Write the **criterion** instead, with two dimensions:

- **Outsourceability** — can the agent do this stably and check it? (drifts ever-leftward as models
  improve)
- **Constitutiveness** — *would its atrophy harm the root of the learner's judgment?* (does **not**
  date)

Only the **high-outsourceability × high-constitutiveness** cell is the true danger zone (FIG L.6): the
agent *can*, and offloading *atrophies the very capability the activity exists to build*. That cell's
border is the stop-line. Produce the explicit two-column boundary:

| Hand to the agent (abundance-able) | Keep with the learner (inside the stop-line) |
|---|---|
| Fact lookup, retrieval, summarizing | The **desirable difficulty** — the productive struggle |
| Drafts, examples, explanations — **only at the *verify* seat, after the learner's attempt** | Problem decomposition, the first hypothesis, the from-scratch reproduction |
| In-paradigm, machine-checkable derivation | Challenge hit-rate, value judgment, intuition, taste, deep thinking |

**The irreducible human judgment node — make it central:** *which difficulty is desirable and must
stay with the human.* If the agent does the struggle, the human doesn't learn — offloading the
desirable difficulty **destroys the very capability the activity exists to build.** Run the criterion
on each candidate step out loud (decompose-the-problem = high-constitutive → keep; generate-boilerplate
= low-constitutive → hand off). This node has teeth: every "keep" entry names *how* it stays with the
human (a self-answer-first move, a removal drill, a delayed-help window), not just an intention.

### Step 3 — Build the scaffold: difficulty engineered into the toolflow
See `references/scaffold-and-resistance.md`. **Resist convenience by *process*, not willpower** —
willpower is scarce and depletes; convenience's default gravity is on 24/7. Build the friction in so
the AI-native default *is* think-before-you-ask. Three pillars (FIG L.5):

1. **Error-and-reflection log** — records *deviation*, not answers (welds in the testing effect).
2. **Same-source, diffable knowledge base** — plain text both learner and agent read; the value is
   not "features" but that you can *diff how your understanding changed* and no mediator stands
   between you and your own cognitive trajectory.
3. **Deliberately inserted difficulty** — the counter-intuitive one, unique to this surface. In a
   convenience-default stack you must *add* friction: substitute **retrieval** for presentation,
   **spacing** for massing, **interleaving** for blocking (Bjork's desirable-difficulty family;
   Roediger & Karpicke's testing effect — grade-II, replicable, AI-independent).

The agent's only legitimate seat in the error-correction loop (FIG L.4) is **④ verify**, entered
*after* the learner's own attempt and self-correction. Move it ahead of ① to draft first and the whole
loop is bypassed — the artifact ships while the learner's structure stays still.

### Step 4 — Stand up the reflection store (反思库)
See `references/artifacts-spec.md`. This is the compounding-context layer — a learning loop that
lives only in prose **does not exist** (M.04). Specify the store as real fields with a real write-back
edge, not a vibe. Minimum entry: a **self-answer field** (the learner's hypothesis, written *before*
asking the agent), the deviation found, *why* it was wrong, a transfer hook to related errors, and a
**next-review date**. The store compounds: by month-three you can diff your mental model and see which
blind spots cleared and which biases held — a capacity no single entry gives, emergent from the whole.

### Step 5 — Build the dashboard (仪表盘): measure capability, not throughput
See `references/dashboard-and-evidence.md`. The dashboard's job is to keep the learner's *self-read
honest*, because the central trap is metacognitive: **recognition fluency is misread by the brain as
reproduction capacity** ("I nodded along, so I learned it"). Track leading indicators that separate
real capability from dependence:
- **Transfer-test pass rate** — can they perform in a *new* context with **no agent present**? (the
  signal, not answer-recall rate)
- **Challenge hit-rate** — how often, and how *correctly*, they push back on the agent. When this
  falls, the metacognitive monitoring loop's first node is corrupted (FIG L.10).
- **Reflection-log return-use rate** — do they actually reread their own errors?
- **Removal-drill result** — pull the AI today; output merely dips (still on the scaffold) vs.
  collapses (crossed into crutch). This *locates the break-point* (FIG L.9), which is never printed
  on the tool.
- **Offloading evidence** — name the synthetic-confidence gauge: METR's "felt faster, measured
  slower" [R19] is the canonical warning that the learner believes they're on the scaffold side while
  already past the break-point.

### Step 6 — Honesty pass: evidence grade + the slow that can't be compressed
See `references/dashboard-and-evidence.md`. **State the stance as the methodology does: grade B — an
evidence-grounded warning plus a falsifiable bet, NOT "AI makes you stupid."** The atrophy hypothesis
(Gerlich 2025 correlational; MIT/Kosmyna preprint; the Google effect) and the redistribution
hypothesis (cognitive resources reallocate upward; the extended mind) read the *same* data oppositely,
and **no longitudinal data yet adjudicates**. So you are not delivering a verdict — you are buying
**low-cost insurance**: the resistance postures (self-answer first, keep the loop, build desirable
difficulty) *lose in neither world* (if atrophy is real they avert it; if redistribution is real they
are exactly the moves that steer resources upward). Frame it as a decision-theory move, not a moral
one. And honor the **value of slow**: consolidation needs physical time (spacing, sleep-stage replay)
the agent **cannot compress** — design the scaffold *along* that time constant, not against it.

## Greenfield vs. transformation — why the logic differs

- **Greenfield builds from the criterion.** No hollowed-out capacity to recover, so the work is
  generative: name the constitutive core, draw the boundary, engineer the difficulty, start small and
  keep going (compounding needs *time* — a grand scaffold abandoned in three days never compounds).
- **Transformation must audit before it rebuilds.** The learner has been "using AI for X" and the
  question is *which constitutive steps got silently offloaded.* The audit is a removal drill on the
  core task; its output is the list of capacities to put back inside the stop-line, then rebuild their
  loop and difficulty. Don't skip the audit — without it you re-automate the very thing that atrophied.

## Depth modules — apply the ones your case triggers

Match weight to the case; a one-week single-skill protocol does not need all of these.

| Module | Apply when |
|---|---|
| Stop-line criterion run on each step | **always** — the offload-boundary is the deciding artifact |
| Reflection-store field spec (self-answer + next-review) | **always** — it's the compounding context |
| Metacognitive-calibration loop | the learner reports "I get it when I read it but can't reproduce it" |
| Evidence-grade honesty (B / falsifiable bet) | any claim about atrophy/benefit — never overclaim either way |
| Desirable-difficulty boundary (Bjork caveat) | a beginner with no base — friction becomes self-harm; dial it down |
| Challenge-hit-rate as in-domain sensitivity | the learner wants "critical thinking" — it can't be a standalone course; it rides on in-domain doing |
| Series seam: trust-but-verify | the learner also ships work — engineering distrusts the *artifact*, learning distrusts its own *dependence* |

State which you triggered and which you skipped, so the depth is a visible choice. **Synthesis over
checklist:** each module earns its place only by sharpening the scarce-judgment / compounding-context
spine for *this* learner.

## Output contract

Deliver an **AI-Native Learning Protocol** written to a file. It must contain, in order:

1. **Scope verdict** — greenfield / transformation / out-of-scope, with rationale. If out-of-scope,
   stop short with the honest redirect.
2. **The capacity** — stated in knowing-how terms + its constitutive core vs. abundance-able shell.
3. **The offload-boundary (交与不交)** — the two-column table, derived by *running the two-dimension
   criterion on each step*, with the desirable-difficulty stop-line marked as the central human node.
4. **The scaffold** — the three pillars instantiated, with the agent pinned to the *verify* seat and
   the friction-into-process design (not willpower).
5. **The reflection store** — concrete fields incl. the **self-answer field** and **next-review date**,
   with the write-back edge named (what gets written, what reads it back).
6. **The dashboard** — the leading indicators (transfer pass-rate, challenge hit-rate, return-use rate,
   removal-drill result) with the synthetic-confidence gauge named.
7. **Honesty pass** — evidence grade B, the falsifiable bet, the value-of-slow time constant.
8. **First move** — the smallest startable version (e.g. "one self-answer move before AI, three
   reflection entries this week").

Then a short **future-trajectory** (1–2 lines): as models get cheaper/stronger, the boundary drifts
left (more enters "AI can do it") but the *criterion* doesn't date; the reflection store compounds into
a personal cognitive moat that a better model cannot hand you. Return the file path + a 2–3 line
summary; don't dump the protocol inline.

## Self-check before delivering
- **Did I gate honestly?** If the ask was "explain X faster" or an org-scale program, did I say so?
- **Is the stop-line central and toothed?** Is the desirable difficulty explicitly *kept with the
  human*, with a named mechanism (self-answer / removal drill / delayed help) — not an intention?
- **Did I keep the agent at the *verify* seat?** If it drafts before the learner's attempt, the loop
  is bypassed — redo Step 3.
- **Does context compound?** Reflection store has real fields + a write-back edge, not prose. A
  self-answer field that's never written is theater.
- **Does the dashboard measure capability, not throughput?** Transfer/reproduction with no AI present —
  not answer-recall or "amount looked up."
- **Did I grade the evidence (B, a bet) instead of claiming "AI makes you stupid"?** Overclaiming
  backfires and isn't supported.
- **Would it survive the redraw-vs-graft test?** State it in one honest line.
- **Does it read as one practitioner's protocol, not a compliance report?** Don't narrate your own
  conformance; let the design carry the rigor.
